# Objective
审查文件上传/下载/删除服务，识别会影响真实用户安全的问题，并输出带证据与修复建议的报告。重点覆盖：扩展名、MIME、内容魔数、大小限制、存储目录、执行权限、路径归一化、随机化文件名、审计日志、日志保留、错误信息泄露。

# Inputs
- 上传/下载/删除相关代码、路由、存储逻辑。
- 配置文件：debug、secret、日志、上传目录、权限、保留周期。
- 可选：运行时错误样例、部署目录权限、Web 服务器配置。

# Procedure
1. 定位所有文件操作入口：upload/download/delete/import/export。
2. 检查上传校验：不得只依赖扩展名；必须同时校验白名单 MIME、内容魔数、文件大小、图片解码/重编码等。
3. 检查文件名处理：用户可控 `filename` 不得直接拼接路径；必须做路径归一化、拒绝 `../`、绝对路径、控制字符，并使用随机存储名。
4. 检查存储位置：上传文件不应直接落在可执行公开目录；确认目录无执行权限，下载经鉴权或间接代理。
5. 检查下载/删除：路径必须限制在安全根目录内，防止任意文件读取/删除。
6. 检查审计：上传、下载、删除等高风险操作应记录用户、时间、IP、文件ID、结果，并配置保留周期。
7. 检查错误响应与配置：禁止返回内部路径、邮箱、token、api_key、SECRET_KEY、stacktrace；生产环境不得 `debug: true`，默认密钥不得为 `changeme`。

# Evidence Protocol
每个发现必须包含：
- 证据：引用具体代码/配置片段或字段名。
- 风险：说明真实用户可能受到的影响。
- 严重性：高/中/低。
- 修复建议：给出可执行改法。
不要仅给抽象建议；无证据不下结论。

# Output Contract
输出 Markdown 报告：
1. 总体结论。
2. 发现列表，按严重性排序。
3. 每项包含：标题、严重性、证据、影响、修复建议。
4. 最后给出优先级修复清单。

# Failure Modes
- 只说“检查扩展名”而遗漏 MIME、魔数、大小。
- 忽略路径穿越、公开目录执行权限或随机化存储名。
- 未检查审计日志与保留周期。
- 未检查 debug、内部路径、token、api_key、stacktrace 泄露。
- 发现没有证据或修复建议不可执行。
