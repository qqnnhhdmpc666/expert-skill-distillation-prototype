# Objective
审查文件上传/下载服务中会影响真实用户安全的问题，重点发现可导致任意文件写入/读取、恶意文件执行、敏感信息泄露、审计缺失的缺陷，并给出可落地修复建议。

# Inputs
- 上传、下载、删除相关代码
- 路由、存储路径、文件名处理逻辑
- 配置文件、运行环境配置
- 日志与审计保留策略
- 错误响应样例或调试配置

# Procedure
1. 定位上传入口：检查是否仅按扩展名判断；同时核查 MIME、文件魔数、大小限制、病毒/恶意内容扫描。
2. 检查文件名处理：用户可控文件名不得直接拼接路径；必须做路径归一化、拒绝 `..`、绝对路径、特殊分隔符，并使用随机存储名。
3. 检查存储位置：上传目录不应允许脚本执行；避免直接暴露在公开目录；下载应通过鉴权与受控映射。
4. 检查下载/删除：确认是否存在路径穿越、越权访问、任意文件读取/删除。
5. 检查响应内容：错误或成功响应不得泄露内部路径、邮箱、token、api_key、SECRET、stacktrace。
6. 检查审计：上传、下载、删除等高风险操作应记录操作者、对象、时间、结果、来源 IP，并配置合理保留周期。
7. 检查配置：禁止生产 `debug: true`；密钥不得为默认值或硬编码。

# Evidence Protocol
每个发现必须包含：
- 证据：引用代码/配置片段或字段名
- 风险：说明真实用户可能受何影响
- 可利用条件：攻击者需要控制什么输入
- 修复建议：给出具体实现方向
不得凭空假设未提供的系统行为；不确定时标为“需确认”。

# Output Contract
输出审查报告，包含：
1. 摘要：总体风险等级
2. 发现列表：按严重度排序
3. 每项发现格式：
   - 标题
   - 严重度
   - 证据
   - 影响
   - 修复建议
4. 正向观察：已满足的安全点，如有
5. 待确认问题：证据不足但需核实的事项

# Failure Modes
- 只说“检查扩展名不安全”但不给 MIME、魔数、大小、执行权限建议
- 忽略路径拼接导致的路径穿越
- 忽略下载接口的任意文件读取风险
- 把调试路径、SECRET、stacktrace 泄露当作低风险
- 没有审计日志和保留周期检查
- 给出泛泛建议而非可执行修复措施
