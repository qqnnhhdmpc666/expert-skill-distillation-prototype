# v0.2 Problem Definition

## 1. 系统真正解决的问题

本项目要解决的不是“如何建一个知识库”，也不是“如何自动写更长的 Skill”。真正的问题是：

> 对于一个要长期执行真实任务的 Agent，如何把专家知识分配到可执行 Skill、运行时可检索知识和经验轨迹三种形态中，并用任务结果持续验证、更新、降级或回滚这些知识形态。

这是一个 **knowledge form and lifecycle control** 问题。

输入可能包括专家文档、代码仓库、规范、版本信息、案例、历史轨迹和外部反馈。系统需要回答两类问题：

1. **编译时**：哪些内容应固化为稳定 how-to，哪些应保留为动态知识，哪些只应作为案例或待验证假设？
2. **运行时**：面对具体任务，Agent 应加载哪些 Skill、是否检索、检索哪些来源、何时停止检索，以及如何记录可复验轨迹？

## 2. 三类长期对象

### Skill

Skill 是稳定、程序化、跨任务可复用的执行策略，包括：

- 目标分解与检查顺序；
- 工具调用流程；
- 输出和证据契约；
- 边界、禁用项与恢复策略；
- 在多个任务上反复成立的经验。

Skill 不应保存易变事实、单个漏洞细节或大量案例全文。

### Retrievable Knowledge

可检索知识是运行时按需访问的事实与上下文，包括：

- 版本、CVE、公告、API/框架文档；
- 仓库符号、调用关系和配置；
- 历史案例与失败恢复记录；
- 长尾异常、环境差异和未稳定结论。

它不等于“向量数据库中的文本块”。它可以来自 BM25、代码搜索、AST/LSP、数据库/API、结构化轨迹索引或长上下文读取。

### Trajectory

轨迹是 Agent 的观察、检索、工具调用、决策、失败、恢复和环境结果的时间序列。轨迹首先是证据，不天然是 Skill。

轨迹可能产生：

- 可迁移模式 -> Skill candidate；
- 特定环境经验 -> case knowledge；
- 失败与反例 -> rejected buffer / limitation；
- 外部结果 -> promotion evidence。

## 3. 目标用户

第一目标用户不是普通安全扫描用户，而是：

- 构建领域 Agent 的研发人员；
- 提供知识和审核边界的领域专家；
- 研究 Skill/RAG/Agent learning 的实验人员；
- 需要审计 Agent 知识来源与更新过程的团队。

安全工程师是第一阶段 vertical adapter 的用户，不是通用 Runtime 的唯一用户。

## 4. 与相邻系统的区别

### 相比普通 RAG

普通 RAG 主要回答“这次生成需要哪些上下文”。本项目还要回答：

- 反复检索到的稳定流程是否应编译为 Skill；
- 检索知识是否过期、污染或只适用于某环境；
- 轨迹是否证明知识真正有任务效用；
- 更新失败时如何拒绝、退休或回滚。

### 相比普通 Skill 库

普通 Skill 库主要保存和路由已有 Skill。本项目还要处理：

- Skill 从何种证据蒸馏而来；
- 哪些内容不应进入 Skill；
- Skill 与动态知识如何协同；
- Skill 变更是否有独立任务收益。

### 相比轨迹学习或经验记忆

轨迹检索通常把历史案例直接放回上下文。本项目增加：

- 轨迹到 Skill / case knowledge 的形态分流；
- 多轨迹归纳与反例对照；
- held-out promotion gate；
- provenance、版本、退休和 rollback。

## 5. 核心研究问题

### RQ1：知识形态决策

对一个知识单元，何时选择：

```text
Skill / Retrieval / Both / Trajectory-only / Discard
```

可检验变量包括稳定性、时效性、程序化程度、跨任务复用、环境依赖、错误代价和可验证性。

### RQ2：知识边际价值

在相同 Agent、模型、预算、任务和 evaluator 下：

- Skill 的边际收益是多少？
- Retrieval 的边际收益是多少？
- 两者是否互补，还是产生上下文冲突？
- 长上下文直接阅读何时优于检索？

### RQ3：轨迹双向更新

哪些轨迹证据足以支持 Skill 更新，哪些只应进入案例库？成功轨迹、高奖励轨迹和高 teaching utility 轨迹是否相同？

### RQ4：稳定流程与动态知识协同

Skill 是否能规定“何时、去哪里、用什么查询、如何验证检索结果”，而不是把检索结果本身固化？

### RQ5：漂移防御

如何防止：

- 错误事实被固化；
- 检索污染触发错误 Skill 修改；
- 同一轨迹同时进入 Skill 和知识库后被重复放大；
- Agent 能力变化使旧 Skill 变成负担；
- verifier 偏差递归强化自身。

## 6. 第一阶段垂直场景

推荐选择防御性代码安全与已知漏洞修复，而不是直接做开放式零日发现。

适合的任务：

- 已知漏洞定位和证据绑定；
- 输入验证、权限控制、配置安全审查；
- 依赖版本与公告确认；
- 安全补丁生成与测试；
- 仓库级调用链和可达性分析。

它能自然拆出三层：

```text
Skill: 稳定审查流程、工具顺序、证据标准、安全边界
Knowledge: CVE/版本/框架文档/仓库结构/历史案例
Trajectory: 搜索、读文件、运行测试、失败与恢复
External result: 测试、补丁、benchmark-native verifier
```

不建议第一阶段做：自动 exploit、真实目标攻击、无授权扫描、以自建 capability label 代替真实漏洞结果。

## 7. 明确不解决什么

- 不构建通用企业知识管理平台；
- 不承诺任意文档都能自动蒸馏为高质量 Skill；
- 不承诺一个 router 能覆盖所有领域；
- 不把 LLM-as-a-judge 当作高风险安全任务的唯一最终裁判；
- 不证明 Skill-only 一定优于 RAG-only；
- 不默认 GraphRAG、多 Agent 或向量数据库是必要条件；
- 不把本地受控安全审查称为真实世界漏洞发现。

## 8. 成功标准

系统级成功不由文档数或内部总分定义，而由以下证据共同支持：

1. 同一任务矩阵中，知识形态选择可被复验；
2. Skill、Retrieval 和 Both 的边际效用可分别测量；
3. 外部任务结果优先于内部诊断分；
4. 轨迹更新能至少在一个独立 held-out family 上产生净收益；
5. negative control、过时知识、检索污染和 scope violation 不退化；
6. 所有更新有来源、版本、访问记录和 rollback pointer。

在这些条件出现前，项目应表述为“混合知识形态与 Skill 生命周期研究原型”，而不是成熟自动知识进化系统。
